#+title: Replace FlexSearch modal with Pagefind components without regressing UX
#+author: Darwin D. Wu
#+status: Done
#+date: <2026-06-10 Wed>

Swap the FlexSearch-powered search modal for [[https://pagefind.app/][Pagefind]],
using Pagefind's Component UI where it can be styled to closely match the
current modal. Make search an explicit opt-in feature because Pagefind requires
a downstream post-build indexing step. When enabled, keep the existing trigger
affordance, Cmd/Ctrl+K shortcut, accessibility expectations, and
=Site.Params.search= toggle intact. The goal is a smaller initial payload and a
simpler index pipeline (no custom Hugo output format) without any user-visible
regression for sites that opt in.

* Background

The current search stack:

- =layouts/partials/search.html= renders the trigger, the modal, and inlines the
  client code. State, focus trap, and keybindings are Alpine.js
  (=searchbar=/=searchterm=/=searchresults=). Cmd/Ctrl+K opens, =Escape= closes,
  Tab cycles through results, =role="dialog"= + =aria-modal=.
- =static/js/flexsearch-0.8.2.bundle.min.js= ships FlexSearch's Document API.
  The index is built client-side at page load from a JSON payload.
- =layouts/_default/list.searchindex.json= emits the payload via a custom Hugo
  =SearchIndex= output format declared in =config/_default/hugo.yaml=. Every
  page's =title=, =summary=, and full =content= is shipped on every page load.
- =demo/hugo.yaml= opts into =SearchIndex= for the home output so the demo site
  exercises search.
- The =search= site param (default =true= today) gates rendering in
  =layouts/partials/nav.html=.

Two problems with the status quo:

1. The whole index downloads on first visit, even for users who never open
   search. Payload grows linearly with the site.
2. The custom Hugo output format and template are theme-specific machinery that
   Pagefind makes unnecessary — Pagefind indexes the rendered HTML in =public/=
   directly.

Pagefind ships a standalone binary (no Node), shards its index, fetches only
the slices a query needs, and includes a Component UI built from web components
that can provide the modal, input, summary, results, and keyboard-hint pieces.
That fits this repo's "no Node toolchain" stance and can preserve the modal's
UX surface if the theme overrides Pagefind's default component styling instead
of accepting the stock widget appearance.

Because this repository is a Hugo theme, the generated Pagefind index is not
owned by the theme when a downstream site imports it. The theme can provide the
search partial, Pagefind metadata/ignore attributes, styling, and documentation,
but each consuming Hugo site must install Pagefind in its own build environment
and run it after Hugo has rendered that site's =public/= directory. The expected
consumer build flow is:

#+begin_src sh
hugo --minify
pagefind --site public --force-language en
#+end_src

That writes =public/pagefind/=, which must be deployed with the rest of
=public/=. A plain =hugo server= run does not produce a fresh Pagefind index;
this repository's =make dev= wraps Hugo's live-reload server with a file watcher
that refreshes =demo/public/pagefind/= after source changes. If a consuming site
enables =Site.Params.search= but does not ship =public/pagefind/=, the component
UI and/or index files will 404. Those sites should either add the post-build
Pagefind step or leave =params.search = false= / unset.

* Decisions

- Add =pagefind= to the Nix dev shell so =nix develop= provides the binary
  alongside Hugo, Tailwind, Go, and =git-cliff=. Match the precedent set by
  =tailwindcss-v4= in [[file:001-remove-npm-dependencies.org][001-remove-npm-dependencies]].
- Change search from default-on to opt-in. Rendering should require an explicit
  =params.search = true=, not =.Site.Params.search | default true=. This avoids
  shipping a broken search trigger for theme consumers who have not installed
  Pagefind or added the required post-Hugo indexing step.
- Prefer Pagefind's Component UI over hand-rolled result rendering. Start with
  =pagefind-modal= and its explicit internal structure
  (=pagefind-modal-header=, =pagefind-input=, =pagefind-modal-body=,
  =pagefind-summary=, =pagefind-results=, and optionally
  =pagefind-keyboard-hints=). Keep the existing theme trigger if
  =pagefind-modal-trigger= cannot be styled to match the current nav button and
  keyboard affordance closely.
- Load the Component UI assets from the generated Pagefind bundle
  (=pagefind-component-ui.js= and =pagefind-component-ui.css=) in a way that
  preserves the current payload goal. The Pagefind search index itself should
  remain lazy via =pagefind-config= without =preload=, so only users who search
  fetch the index shards. If first-open latency is unacceptable in manual
  testing, allow eager loading of the component shell while keeping the index
  lazy.
- Compute the Pagefind bundle URL through Hugo (for example
  ="/pagefind/" | relURL=), not as a hard-coded root path. The theme and demo
  can be served under a subpath, and =pagefind-config='s =bundle-path= must
  match that base path before the components initialize.
- Match the current search design closely: same trigger placement and label,
  same Cmd/Ctrl+K affordance, same overlay/backdrop feel, same modal width and
  top offset, same input density, same result spacing, same empty/loading
  states, same light/dark theme behavior, and no stock Pagefind visual reset
  bleeding into the theme. Pagefind's CSS variables are the first styling
  layer; add targeted CSS only where variables are insufficient.
- Preserve the current keyboard and accessibility behavior at the UX level:
  Cmd/Ctrl+K opens, =Escape= closes, focus lands in the input, focus remains
  trapped while open, result navigation is keyboard usable, and screen readers
  get a real dialog/search experience. Pagefind's modal component may replace
  Alpine's focus trap if manual verification confirms parity.
- Preserve =SEARCH_LIMIT = 5= behavior via Pagefind component configuration
  where possible, or via the closest Component UI option/attribute available in
  the pinned Pagefind version.
- Expose page =.Type= via =data-pagefind-meta="type"= on the page wrapper so the
  existing icon switch in =search.html:67-94= keeps working without a template
  rewrite.
- Let Pagefind's results component render highlighted excerpts when using the
  Component UI. If preserving the current type icon switch requires a custom
  result template/component, render Pagefind-highlighted excerpts as sanitized
  HTML and treat =meta= values as untrusted text.
- Delete the =SearchIndex= output format, =list.searchindex.json=, and the
  bundled =flexsearch-*.bundle.min.js=. No backwards-compat shim — the theme
  has one consumer pattern and Pagefind replaces it cleanly.
- Run Pagefind with =--force-language en= after =hugo --minify= so the index
  language is stable even if a rendered page lacks or changes =html lang=. The
  generated =public/pagefind/= directory is build output and is already covered
  by the existing =public/= ignore rule.
- Treat Pagefind indexing as a downstream build contract for theme consumers.
  The theme cannot automatically build an importing site's index; documentation
  must say that consumers need a Pagefind binary and a post-Hugo step that
  deploys =public/pagefind/= alongside the site.

* Notes from =../emacs-docs=

=../emacs-docs= uses Pagefind successfully, but for a larger documentation site:

- Its =scripts/build-pagefind.mjs= creates one bundle per manual/version with
  =--glob=, =--output-subdir=, and =--force-language en=. This theme should not
  copy the multi-bundle complexity, but it should copy the explicit language and
  output-path discipline.
- Its templates put =data-pagefind-body= only on the real article body and
  =data-pagefind-ignore= on modal, sidebar, breadcrumbs, related links, page
  navigation, and footer chrome. In this theme, adding =data-pagefind-body=
  means every intended searchable layout must receive it; otherwise Pagefind
  will skip pages without that attribute.
- Its search UI uses Pagefind's component web UI. This plan should now copy
  that direction, but without copying the visual treatment wholesale: initialize
  or load the component shell at the right time, focus after it is ready, ensure
  the modal itself is ignored by the index, and override component styles so the
  result is visually close to the current modal.
- Its source filters use =data-pagefind-filter= and component events. Filters
  are unnecessary for this migration, but a future section/type filter could use
  the same pattern if the theme later wants faceted search.

* Out of scope

- Pagefind's older Default UI (=pagefind-ui.js=). Prefer the newer Component UI
  (=pagefind-component-ui.js=) because it supports composing the modal from
  smaller web components while keeping styling close to the existing theme.
- Multi-language indexing. The current theme is single-language; Pagefind
  supports =data-pagefind-language= but that's a follow-up if/when the theme
  grows i18n.
- Search analytics, filters, or faceting. Pagefind supports filters via
  =data-pagefind-filter=, but the existing UX has none — adding them is a
  design change, not a migration.
- Replacing Alpine.js with a different reactive layer. The modal already works;
  Alpine isn't the bottleneck.
